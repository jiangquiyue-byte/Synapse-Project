# Synapse 每日开发日志（2026-03-29）

## 今日目标

围绕开发蓝图中 **M5 生产化升级** 与后续交互补齐任务，推进以下方向：

1. 将后端从内存态迁移到可持久化的数据库抽象层，为 **PostgreSQL + pgvector** 落地做好结构准备。
2. 将聊天、Agent、工作流、用户配置与会话列表逐步接入新的持久化接口，确保刷新或服务重启后状态可恢复。
3. 将 `memory.py` 从占位接口升级为真实的跨会话记忆检索能力。
4. 将原有 Agent 级流式输出升级为 **Token 级逐字流式输出**。
5. 将前端从原三页面扩展为五页面，新增 **记忆中心** 与 **工作流市场**，并补齐 **导出对话入口** 与 **Tavily 联网搜索开关**。
6. 将今天全部实施内容同步到项目内日志，方便下一位继续接手。

## 今日已完成内容

### 1. 生产级持久化底座已落地

后端数据库抽象层已从纯内存方案升级为 **SQLAlchemy 异步持久化实现**，并支持：

- `PostgreSQL + pgvector` 作为生产目标方案；
- `SQLite` 作为本地开发降级方案；
- 会话、消息、Agent、工作流、用户配置、文档分块、记忆记录等统一存储模型。

本轮已更新的关键文件包括：

- `backend/app/models/database.py`
- `backend/app/core/config.py`
- `backend/app/main.py`
- `backend/app/routers/state.py`
- `backend/requirements.txt`
- `backend/api/requirements.txt`

### 2. 会话、Agent、用户配置已接入后端持久化链路

后端新增并暴露了状态同步路由，前端 API 与 Zustand store 也已升级，当前可以通过后端接口读取：

- 会话列表；
- 当前会话历史；
- 用户配置；
- Agent 列表与工作流数据。

本地验证结果表明：

- `GET /api/state/sessions` 已可返回实际持久化会话；
- `GET /api/chat/history/{session_id}` 已可读取历史消息；
- 聊天消息在流式结束后已写回持久化层。

### 3. 跨会话记忆系统已替换占位实现

`memory.py` 已不再是纯占位接口，而是通过新的 `memory_service.py` 提供：

- 按会话查看记忆；
- 跨会话语义检索；
- 上下文预览拼装；
- 中文场景下的 CJK 关键词降级召回。

本地验证结论：

- 记忆写入已生效；
- `GET /api/memory?limit=5` 可返回真实历史记忆；
- `GET /api/memory/search?query=持久化&limit=3` 已能跨会话返回命中结果；
- 当前本地环境由于嵌入模型接口返回 `404`，所以暂时走 **关键词/CJK 分词降级召回**；
- 一旦生产环境中的 embedding 接口可用，将自动切换到向量召回。

### 4. Token 级流式输出已打通

后端编排器与聊天流式路由已改造完成，前端 SSE 客户端与消息状态更新逻辑也已同步升级。

本地验证显示：

- `/api/chat/stream` 已可稳定产出 `event: token` 事件；
- 单条回复按字符持续输出；
- 最终回答仍会被持久化并可从历史接口回放；
- 当前流式事件链路已经满足“逐字打字机效果”的基础要求。

### 5. 前端已从三页面扩展到五页面

新增页面已创建并接入现有导航：

- `mobile/app/(tabs)/memory.tsx`
- `mobile/app/(tabs)/workflows.tsx`

同时已完成：

- 五页面导航接入；
- 1px 黑线极简图标风格延续；
- 聊天页导出对话入口接入；
- 设置页 Tavily 联网搜索开关接入；
- 前端图标统一为纯代码 SVG 方案。

### 6. UI 与品牌系统延续完成

在之前完成的基础上，本轮保持并延续了以下视觉规范：

- 所有功能性图标继续使用纯代码 SVG；
- `SynapsePulse` 作为全站 Loading 品牌动画组件继续保留；

## 当前遗留问题 / 风险提示

### 1. 生产环境 Tavily 仍未真正启用

当前健康检查返回：

> `"tavily_enabled": false`

这说明当前运行环境尚未注入 `TAVILY_API_KEY`。前端开关与后端接线已经具备，但生产可用性仍取决于部署平台环境变量配置。

### 2. PostgreSQL + pgvector 已完成代码接入，但当前本地仍运行在 SQLite 降级态

当前健康检查返回：

> `"database": "sqlite-dev"`

说明数据库抽象层已支持生产数据库，但当前验证环境尚未连接真实 PostgreSQL。下一位接手时应优先完成：

1. 准备 PostgreSQL 实例；
2. 安装/启用 `pgvector` 扩展；
3. 配置 `DATABASE_URL`；
4. 重新验证会话、记忆、工作流与导出链路；
5. 重点验证 embedding + pgvector 路径，而非仅关键词降级路径。

### 3. 记忆系统当前主要依赖关键词降级召回

日志中可见：

> `Memory embedding generation failed: Error code: 404 - {'status': 'not found'}`

这表示当前 embedding 接口不可用，因此虽然“跨会话记忆”已经可用，但此刻主要依赖：

- 关键词匹配；
- 中文 CJK 分词；
- 文本召回降级逻辑。

生产环境中需进一步确认：

- `OPENAI_BASE_URL` 是否同时支持 embeddings；
- `EMBEDDING_MODEL` 是否有效；
- 若兼容网关不支持 embedding，需要改成可用服务或做专门路由分流。

### 4. 仓库当前仍存在未提交改动

今天的所有开发已落盘到仓库工作区，但尚未完成最终提交与推送。主要改动集中在：

- 后端数据库、记忆、聊天、导出、工作流、状态路由；
- 前端五页面、SSE、状态层、聊天页、设置页；
- 图标库与 Loading 品牌组件；
- 项目文档与交接文件。

下一步应在确认文档更新完成后执行：
